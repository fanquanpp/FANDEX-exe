# 安全策略

本文件定义 FANDEX 仓库的安全漏洞报告流程、响应时间承诺、披露窗口与适用范围。任何使用者发现潜在安全问题，请按本流程上报，切勿直接公开披露或在 Issue 中描述漏洞细节。

## 支持版本

本仓库当前仅对以下版本提供安全支持：

| 版本 | 状态 | 支持范围 |
| :--- | :--- | :--- |
| 3.x | **支持** | `main` 分支最新代码与官方 Release |
| 2.x | 维护中 | 仅严重安全漏洞修复 |
| 1.x | 不再支持 | 历史归档，不接受安全报告 |

> 建议使用者始终使用最新稳定版本。历史版本中的已知漏洞不保证修复。

## 安全策略

### Tauri 2 安全模型

本仓库桌面端基于 Tauri 2，采用以下安全机制：

| 机制 | 配置 | 作用 |
| :--- | :--- | :--- |
| **Capabilities 最小权限** | `src-tauri/capabilities/` 仅声明应用必需的权限 | 限制前端对原生 API 的访问范围 |
| **CSP 严格策略** | Tauri 配置中启用严格 Content-Security-Policy | 阻止内联脚本与跨域资源注入 |
| **Updater 签名验证** | Tauri updater 启用签名校验 | 防止下载与安装被篡改的更新包 |
| **Context Isolation** | 默认启用 | 隔离 WebView 与 Rust 后端的执行上下文 |
| **禁用 remote API** | Tauri 2 默认不暴露远程 API | 所有 IPC 通过 `#[tauri::command]` 显式声明 |

### Web Worker 沙箱

- 所有重量级计算（搜索索引、术语高亮、知识图谱渲染）在 Web Worker 中执行
- Worker 通过 `postMessage` 与主线程通信，无直接 DOM 访问权限
- Worker 脚本通过严格类型声明（`apps/web/src/workers/types.ts`）约束消息协议

### 内容安全策略（CSP）

Web 端通过 HTTP 响应头与 `<meta>` 标签双重声明 CSP：

- `default-src 'self'`：仅允许同源资源
- `script-src 'self'`：禁止内联脚本与外部 JS
- `style-src 'self' 'unsafe-inline'`：允许 Tailwind 内联样式（必需）
- `img-src 'self' data: blob:`：允许 data URL 与 blob URL 图片
- `connect-src 'self'`：禁止外部 API 调用（除 GitHub Pages CDN）
- `worker-src 'self' blob:`：允许 blob URL Worker

### 数据持久化安全

- 学习进度与笔记通过 IndexedDB 持久化，仅存储在用户本地设备
- 不收集、不上传任何用户学习数据
- 无远程账户系统，无身份认证
- PWA Service Worker 仅缓存静态资源，不缓存用户数据

## 报告漏洞

若你发现安全漏洞或潜在的安全风险，请通过以下方式私密上报，**不要在公开 Issue 列表中创建漏洞相关 Issue**，也不要在 PR、讨论区或社交平台公开漏洞细节：

- 优先邮件：fanquanpangpiing@163.com
- 邮件主题请使用 `[SECURITY] FANDEX-exe - <简短问题描述>` 格式
- 邮件正文请尽量包含以下信息：
  - 受影响的版本与分支（默认聚焦 `main` 分支最新代码）
  - 漏洞复现步骤（如涉及桌面端，请说明操作系统与 Tauri 版本）
  - 影响范围与可能的攻击场景
  - 建议的修复方向（可选）
  - 报告者希望使用的署名信息（用于后续致谢，可匿名）

收到报告后，维护者将通过同一邮箱与报告者沟通后续进展，避免漏洞细节外泄。

也可通过 [GitHub Security Advisory](https://github.com/fanquanpp/FANDEX-exe/security/advisories/new) 私密提交安全报告。

## 响应时间

维护者按以下时间窗口响应安全报告：

- **48 小时内**：确认收到报告，并向报告者回执
- **7 天内**：完成初步评估，给出是否认定为安全漏洞、影响等级（严重 / 高 / 中 / 低）与初步处置计划
- **30 天内**：对于被认定为有效漏洞的报告，给出修复方案或临时缓解措施
- **90 天内**：完成修复并通过协调披露流程公开漏洞信息与修复致谢

如遇节假日或特殊情况导致响应延迟，维护者将在以上时间窗口内向报告者主动说明延迟原因与预计响应时间。

## 披露窗口

本仓库采用 **90 天协调披露**（Coordinated Disclosure）策略：

- 报告者提交有效漏洞后，维护者在 90 天内完成修复并发布修复版本
- 在漏洞修复版本发布前，维护者与报告者共同遵守披露静默期，不对外公开漏洞细节、复现步骤与受影响代码路径
- 修复版本发布后，维护者将通过 Release Notes 或安全公告公开漏洞概述、影响范围与修复版本信息，并对报告者予以致谢（除非报告者要求匿名）
- 在以下情形下，披露窗口可由双方协商调整：
  - 漏洞影响范围极为广泛，需要更长修复周期
  - 报告者主动同意延长静默期
  - 漏洞已被第三方公开披露，无可继续协调的窗口

## 适用范围

本安全策略仅适用于以下范围：

- 本仓库 `main` 分支上的最新代码
- 本仓库官方发布的 Release 版本中已知的受影响代码路径
- 本仓库默认配置下运行的行为；若使用者自行修改配置、关闭安全选项或对接未授权的第三方依赖所引发的安全问题，不在本策略覆盖范围

对于历史分支、已被废弃的实验性代码或仓库历史归档版本中的漏洞，维护者将视严重程度决定是否处置，不强制纳入响应时间承诺。

## 不在范围内的行为

以下行为不属于本仓库鼓励的安全测试范围，相关报告将不予受理，必要时维护者将保留采取进一步措施的权利：

- 对本仓库线上服务（GitHub Pages）进行分布式拒绝服务（DDoS / DoS）攻击
- 对本仓库维护者、贡献者或用户进行社会工程学攻击、钓鱼攻击或身份冒充
- 针对本仓库物理基础设施、办公网络或个人设备的物理安全测试
- 大规模自动化扫描、爆破、fuzzing 给线上服务造成明显负担的行为
- 已知未修补的第三方依赖 CVE 漏洞，且本仓库已在依赖配置中提供最新可用版本，但使用者未及时升级所致的问题
- 利用本仓库代码在非授权环境部署后，因部署配置不当引发的安全问题
- 在漏洞未修复前，公开披露漏洞细节或对外提供可复现的 PoC

## 致谢

对于提交有效安全漏洞的报告者，本仓库将在漏洞修复公告中予以致谢：

- 默认使用报告者邮件中提供的署名信息（GitHub 用户名、姓名或昵称）
- 若报告者要求匿名，将仅以 `匿名报告者` 字样致谢，不公开任何个人信息
- 致谢信息将随修复版本 Release Notes 一并发布，发布后不再撤回，除非报告者提出明确且合理的撤回请求

维护者感谢每一位负责任披露安全问题的贡献者，并承诺在沟通过程中保持专业、尊重与透明。
