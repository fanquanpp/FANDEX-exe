## DevOps 术语表

## B

| 术语     | 英文                  | 释义                                                       |
| -------- | --------------------- | ---------------------------------------------------------- |
| 编排     | Orchestration         | 自动化管理多个容器和服务的部署、扩缩、网络和生命周期的过程 |
| 蓝绿部署 | Blue-Green Deployment | 通过维护两套相同生产环境（蓝和绿）实现零停机发布的部署策略 |

## C

| 术语     | 英文                   | 释义                                                               |
| -------- | ---------------------- | ------------------------------------------------------------------ |
| 容器     | Container              | 将应用及其依赖打包为轻量级、可移植的运行单元，共享宿主操作系统内核 |
| 持续集成 | Continuous Integration | 开发者频繁将代码合并到主分支，每次合并自动触发构建和测试的实践     |
| 持续部署 | Continuous Deployment  | 代码通过所有测试后自动部署到生产环境的流水线，无需人工干预         |
| 持续交付 | Continuous Delivery    | 代码通过测试后自动部署到预生产环境，随时可手动发布到生产环境       |

## D

| 术语       | 英文            | 释义                                                         |
| ---------- | --------------- | ------------------------------------------------------------ |
| 端到端测试 | End-to-End Test | 模拟真实用户操作流程，验证整个系统从入口到出口的完整功能链路 |

## F

| 术语     | 英文         | 释义                                                                        |
| -------- | ------------ | --------------------------------------------------------------------------- |
| 服务网格 | Service Mesh | 处理服务间通信的基础设施层，以 Sidecar 代理实现流量管理、可观测性和安全策略 |

## G

| 术语     | 英文           | 释义                                                                         |
| -------- | -------------- | ---------------------------------------------------------------------------- |
| GitOps   | GitOps         | 以 Git 仓库作为声明式基础设施和应用配置的唯一真实来源，通过 Git 操作驱动部署 |
| 滚动更新 | Rolling Update | 逐步替换旧版本实例为新版本的部署策略，保持服务持续可用                       |

## H

| 术语 | 英文     | 释义                                                           |
| ---- | -------- | -------------------------------------------------------------- |
| Helm | Helm     | Kubernetes 的包管理器，使用 Chart 模板定义、安装和升级复杂应用 |
| 回滚 | Rollback | 将部署恢复到先前已知正常版本的操作，是故障恢复的关键手段       |

## J

| 术语           | 英文                   | 释义                                                                    |
| -------------- | ---------------------- | ----------------------------------------------------------------------- |
| 基础设施即代码 | Infrastructure as Code | 使用代码而非手动流程来管理和配置基础设施的实践，支持版本控制和自动化    |
| 金丝雀发布     | Canary Release         | 将新版本先部署到少量实例进行验证，确认无问题后再逐步扩大范围的发布策略  |
| 监控           | Monitoring             | 持续采集和展示系统指标（CPU、内存、延迟等）的过程，用于了解系统运行状态 |

## K

| 术语     | 英文          | 释义                                                 |
| -------- | ------------- | ---------------------------------------------------- |
| 可观测性 | Observability | 通过日志、指标和链路追踪三支柱理解系统内部状态的能力 |

## L

| 术语     | 英文     | 释义                                                          |
| -------- | -------- | ------------------------------------------------------------- |
| 流水线   | Pipeline | 定义从代码提交到部署的自动化步骤序列，是 CI/CD 的核心实现方式 |
| 链路追踪 | Tracing  | 跟踪请求在分布式系统中经过各服务的完整路径，用于定位性能瓶颈  |

## M

| 术语     | 英文      | 释义                                                      |
| -------- | --------- | --------------------------------------------------------- |
| 命名空间 | Namespace | Kubernetes 中用于资源隔离和组织的逻辑分区，支持多租户环境 |

## P

| 术语 | 英文  | 释义                                                                  |
| ---- | ----- | --------------------------------------------------------------------- |
| 探针 | Probe | Kubernetes 中检测容器健康状态的机制，包括存活探针、就绪探针和启动探针 |

## Q

| 术语     | 英文              | 释义                                                   |
| -------- | ----------------- | ------------------------------------------------------ |
| 混沌工程 | Chaos Engineering | 主动向系统中注入故障以验证系统韧性和恢复能力的实验方法 |

## S

| 术语 | 英文                    | 释义                                                            |
| ---- | ----------------------- | --------------------------------------------------------------- |
| SLA  | Service Level Agreement | 服务提供商与客户之间定义的正式服务质量承诺，包含违约赔偿条款    |
| SLI  | Service Level Indicator | 衡量服务质量的具体量化指标，如可用性百分比、请求延迟百分位数    |
| SLO  | Service Level Objective | 团队为 SLI 设定的内部目标值，如 99.9% 可用性或 P99 延迟 < 200ms |

## T

| 术语     | 英文         | 释义                                                                  |
| -------- | ------------ | --------------------------------------------------------------------- |
| 特征开关 | Feature Flag | 在不修改代码的情况下动态启用或禁用功能的配置机制，支持灰度和 A/B 测试 |

## W

| 术语   | 英文         | 释义                                                 |
| ------ | ------------ | ---------------------------------------------------- |
| 微服务 | Microservice | 将应用拆分为多个独立部署、松耦合的小型服务的架构风格 |

## Y

| 术语     | 英文           | 释义                                                           |
| -------- | -------------- | -------------------------------------------------------------- |
| 压力测试 | Stress Testing | 在超出正常负载条件下测试系统行为，验证系统的极限容量和恢复能力 |

## Z

| 术语 | 英文     | 释义                                                                 |
| ---- | -------- | -------------------------------------------------------------------- |
| 告警 | Alerting | 当监控指标超过预设阈值时自动发送通知的机制，确保运维团队及时响应异常 |
